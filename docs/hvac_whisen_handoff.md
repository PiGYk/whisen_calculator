
# HVAC Calc / Whisen Quote — handoff document

## 1. Навіщо цей документ
Цей документ потрібен як повний handoff-контекст для іншої LLM або нового чату, щоб модель одразу розуміла:
- що вже налаштовано;
- де лежать файли;
- у якому середовищі ведеться розробка;
- що вже працює;
- куди рухати проєкт далі;
- які артефакти треба віддавати на кожній фазі.

Головний принцип: **не робити новий проєкт з нуля**, а продовжувати поточний Flutter-проєкт і розвивати його в правильній архітектурі.

---

## 2. Поточна ідея продукту

Проєкт має розвиватися як **одна кодова база з двома продуктовими режимами**:

### Режим A. Quick Calc
Швидкий HVAC-калькулятор для монтажників / менеджерів:
- попередній cooling/heating calculation;
- базове енергоспоживання;
- збереження проєктів;
- без PDF;
- швидкий сценарій використання “ввів → отримав результат”.

### Режим B. Whisen Quote
Більш серйозний режим для Whisen:
- використовує той самий калькулятор як ядро;
- підбирає конкретне обладнання **LG + AUX**;
- генерує красиве **PDF-КП**;
- включає брендований контент, переваги серій, технічні характеристики, картинки;
- орієнтований на менеджерів і продажі.

**Важливо:** це не два окремі незалежні застосунки з нуля, а **одна кодова база, спільне ядро, різні модулі / режими / збірки**.

## 9. Правильна цільова архітектура

Проєкт потрібно будувати не “по екранах”, а по **логічних модулях**.

### Модуль 1. Calculator Core
Відповідає тільки за:
- вхідні параметри;
- формули;
- результат розрахунку;
- енергоспоживання;
- базові recommendation classes.

Цей модуль **не має знати нічого** про PDF, бренди, LG, AUX.

### Модуль 2. Equipment Catalog
Окремий шар з даними про техніку:
- бренди;
- серії;
- моделі;
- технічні параметри;
- ключові переваги;
- картинки;
- технологічні блоки для КП.

Ці дані бажано тримати в `assets/json/` і `assets/images/`, а не хардкодити в Dart.

### Модуль 3. Selection Engine
Бере результат калькулятора і повертає конкретні рекомендовані варіанти:
- Standard
- Optimal
- Premium

Тобто перетворює “потрібно 3.5 кВт” у “ось конкретні LG/AUX моделі”.

### Модуль 4. Proposal Builder
Готує внутрішню структуру комерційної пропозиції:
- дані клієнта;
- дані об’єкта;
- підібране обладнання;
- технічні характеристики;
- переваги;
- енергоспоживання;
- службові примітки / дисклеймери.

### Модуль 5. PDF Export
Окремий модуль, який бере вже готовий `ProposalData` і генерує PDF.  
**Не починати з PDF раніше, ніж будуть готові Calculator Core + Catalog + Selection + ProposalData.**

## 12. Принципи, яких інша LLM повинна дотримуватись

1. **Не починати з нуля**, а розвивати поточний проєкт.
2. **Не зливати все в один `main.dart`.**
3. **Не лізти в PDF раніше часу.**
4. **Не змішувати калькулятор, каталог і PDF в одному модулі.**
5. **Не видаляти старий функціонал під час додавання нового.**
6. Якщо додається нова функція, треба зберігати:
   - існуючий розрахунок;
   - збереження проєктів;
   - список проєктів;
   - перегляд деталей.
7. На кожному етапі віддавати **повні робочі файли**, а не уривки без контексту.
8. На кожному етапі давати:
   - точний список файлів;
   - куди який файл вставляти;
   - які команди запускати;
   - що перевірити руками.
9. Якщо змінюється структура проєкту, LLM повинна окремо написати:
   - що створити;
   - що видалити;
   - що перенести;
   - які імпорти зміняться.
10. Усі рішення мають бути орієнтовані на:
    - чистий UX;
    - акуратний темний технічний стиль;
    - юзабельність;
    - реальне застосування в продажах / монтажі.

## 17. Підсумок для нової LLM

Потрібно продовжити вже існуючий Flutter-проєкт `hvac_calc_pro` на CachyOS.  
Поточне ядро калькулятора вже працює і вже має локальне збереження.  
Далі проєкт треба розвивати як **одну кодову базу** з двома режимами:

- **Quick Calc** — швидкий HVAC-калькулятор без PDF;
- **Whisen Quote** — брендований підбір LG/AUX з генерацією PDF-КП.

Правильний порядок розвитку:
1. refactor + UI polish + release-ready Quick Calc;
2. catalog + selection engine;
3. proposal builder;
4. PDF export + release polish.

Головне правило: **не намагатися робити все одразу**, не руйнувати існуючий функціонал і на кожній фазі віддавати повний, чіткий, придатний до вставки та запуску результат.
